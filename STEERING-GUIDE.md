# Steering Guide

> **What steering is:** Files that tell the agent how to behave in a project — stack, specs, design, session purpose — without being explicit instructions. The agent reads them and adapts automatically.  
> **Why it matters:** No more "what stack?" or "do we have a spec?" — the agent already knows.

---

## Steering Files

| File | Severity | Steers What | Created By |
|---|---|---|---|
| `STACK_CONFIG.md` | 🔴 BLOCKING | Stack defaults for ALL platform skills (Rule 5) | `architecture-analysis` or manually |
| `SPEC.md` | 🟡 HIGH | What to build, acceptance criteria, boundaries | `spec-driven-development` |
| `HEALTH-CHECK.md` | 🟡 HIGH | Codebase health state, known issues, risks | `project-health-check` |
| `PROGRESS_STATUS.md` | 🟡 HIGH | Project state, skill inventory, planned work | manual (verified by `validate-skill-table.sh`) |
| `design/DESIGN-LOCK.md` | 🔵 MEDIUM | Visual tokens, approved direction | `engineering-fundamentals` Phase 2 |
| `.sessionrc` | ⚪ INFO | Session purpose, skill weighting | `init-agents` |

---

## How Steering Works

1. **Rule 0b scans** on session start — finds all steering files, reads them.
2. **Agent adapts** — stack from STACK_CONFIG.md, goals from SPEC.md, health from HEALTH-CHECK.md.
3. **No guesswork** — Agent never asks "what stack?" or "do we have tests?" if steering files exist.
4. **Severity** — BLOCKING files stop the agent until resolved. INFO files are optional.

---

## Creating a Steering File

- **STACK_CONFIG.md:** Copy `STACK_CONFIG_TEMPLATE.md` in repo root. Fill sections for your stack. Delete unused sections.
- **SPEC.md:** Run `spec-driven-development` pipeline. It generates SPEC.md automatically.
- **HEALTH-CHECK.md:** Run `project-health-check`. It creates HEALTH-CHECK.md automatically.
- **PROGRESS_STATUS.md:** Manually maintained. Verified automatically by `validate-skill-table.sh` in pre-commit hook. Run `bash scripts/validate-skill-table.sh` to check accuracy.
- **DESIGN-LOCK.md:** Generated by `engineering-fundamentals` Phase 2C after visual approval.
- **.sessionrc:** Created by `init-agents`. Not git-tracked.

---

## Lifecycle

```
START → Rule 0b scans → Read all steering files → Present summary to user
     → User "continuamos" → Agent adapts behavior per files
     → Changes during session → Update affected steering files BEFORE code
     → Commit — steering files are part of the repo (except .sessionrc)
```

---

## Anti-Patterns

- Agent ignores STACK_CONFIG.md and uses defaults → violates Rule 5.
- Agent skips steering scan at session start → violates Rule 0b.
- Steering files not committed alongside code → loss of context for next agent.
- `.sessionrc` committed to git → contains user-specific session state, not project config.
